What is SOLON?

SOLON Slide Studio automates the creation of data-driven HTML slide presentations. Unlike simple templating, SOLON combines:

  • Template Management: Upload an HTML template with defined content zones
  • Data-Driven Content: Provide structured data (spreadsheets, documents) to the AI
  • AI-Powered Generation: Multi-agent AI orchestration creates custom content per slide
  • Presentation Publishing: Build hierarchical slide structures and export as standalone HTML
  • Git-Based Collaboration: All projects, flows, and exports live in version control
Key Insight: SOLON was created to automate the feature catalog workflow — turning manual slide creation from hours of work into a 15–20 minute agentic AI process. It can be used for recurring reports, roadmap catalogs, proposal decks, and any templated multi-slide workflow.

Core Concepts: Projects & Flows

Projects

A project is a Git repository containing one or more flows, AI context documents, and published presentations. Projects are stored in two locations:

  • Shared: Tracked in Git; team members can clone and collaborate
  • Private: Git-ignored; local-only, useful for drafts or experiments

You can convert a project between shared and private at any time. The app automatically manages .gitignore.

Flows

A flow is a complete, reusable workflow for a single template. It contains:

  • Template HTML: Your uploaded HTML template with content zones defined
  • Zone Selections: Which HTML elements are content placeholders (and their AI instructions)
  • Filters & Grouping: Rules to slice the context data into per-slide instances
  • Generations: History of all AI-generated content rounds
  • Exports: Packaged sets of generated slides ready to publish

All flow state is persisted in flow.json so you can return to it later and regenerate or modify.

Folder Structure on Disk

server/projects/ ├── shared/ # Git-tracked projects │ └── <projectName>/ │ ├── project.json # Project metadata │ ├── AI Context/ # Source docs (xlsx, txt, md, pdf, docx) │ ├── flows/ │ │ └── <flowId>/ │ │ ├── flow.json # All flow state │ │ ├── template.html # Your HTML template │ │ ├── output-*.html # Generated outputs │ │ └── exports/ │ │ └── <exportId>/ │ │ └── <slide>.html # Individual slides │ └── presentations/ # Published presentations └── private/ # Git-ignored projects └── <projectName>/ └── (same structure as shared)

HTML Templates

What is a Template?

A template is a well-formed HTML file (single document) that defines the structure of one or more slides. It can include:

  • Multiple <section> tags, each representing a slide
  • Static HTML (header, footer, styling, navigation)
  • Placeholder <div>, <p>, or <span> elements where content will be inserted
  • CSS styles and responsive layouts
  • JavaScript for interactivity (e.g., sortable tables, collapsible sections)

Defining Content Zones

In SOLON, you select which HTML elements should receive AI-generated content. These are called "zones." For each zone you:

  1. Click the element in the template preview
  2. Assign it a semantic key (e.g., "initiative_name", "description", "estimate")
  3. Optionally provide an AI prompt to guide generation
  4. Mark it as auto-generate (so AI fills it) or manual (you fill it later)

Zones are identified by CSS selectors (e.g., div.header or #summary-text), so your template must have distinguishable elements.

Multi-Slide Templates

If your template has multiple <section> tags, SOLON treats each as a different "slide" within the flow. You define zones for each slide independently. This lets you create heterogeneous flows with different slide types in the same template.

Template Metadata (Optional)

You can embed AI instructions directly in the HTML template using a <meta> tag:

<head> <meta name="ai-instructions" content="... instructions for the AI ..."> </head>

These instructions appear as the default context when generating content for this flow.

Zones & Content Binding

How Zones Work

A zone is a mapping between an HTML element and a piece of content. When you:

  1. Generate content via AI, the AI produces JSON with keys matching your zone names
  2. Apply the JSON, SOLON replaces the zone's HTML text with the AI-generated value
  3. Preview the result in the browser before exporting

SOLON supports both text zones (simple <p>, <span>, etc.) and block zones (entire <div> or <section> fragments).

Zone Features

  • Auto-Discovery: SOLON scans your template and suggests common placeholders (headings, paragraphs, divs)
  • Custom Prompts: For each zone, provide a custom AI instruction (e.g., "Summarize in 50 words")
  • Character Limits: Optionally restrict zone content length (useful for headings)
  • Repeatable Zones: If a zone appears across multiple slide instances, it's marked as repeatable
  • Full-Slide Generation: Toggle to generate all zones on a slide at once (uses template instructions) vs. zone by zone

Inline Editing

In the Preview step, you can click on any generated slide content and edit it directly in the browser. Changes are saved to the output file immediately, so you can fine-tune AI output without re-generating.

Exports & Slide Packaging

What is an Export?

An export is a versioned, packaged set of slides generated from a single AI generation round. Each export contains:

  • Slide Files: Individual, self-contained HTML files (each is a complete webpage)
  • Metadata: Slide titles, creation timestamp, generation IDs
  • Project Index: project.json listing all slides for the viewer

Exports are immutable once created, so you can safely iterate on future generations without losing prior work.

Creating Exports

After you've generated content and previewed it, you create an export in the Metadata step. SOLON:

  1. Extracts all generated <section> elements from the output HTML
  2. Wraps each section in a complete HTML document (with the template's head, scripts, styles)
  3. Infers slide titles from headings (<h1>, <h2>, etc.) or lets you name them
  4. Saves everything to exports/<exportId>/

Using Exports

Exports serve as building blocks for presentations. You can:

  • Browse all exports across your project and hand-pick slides
  • Combine slides from different flows and template types into a single presentation
  • Rename, reorder, or delete slides before publishing
  • Publish a structured presentation with navigation (see Publishing section)

How AI Generation Works

Overview

SOLON uses a multi-agent AI orchestration pipeline. Instead of generating all slides in one prompt, it:

  1. Plans: Reads your context, identifies how many slide instances to generate, and extracts relevant data for each
  2. Orchestrates: Spawns one AI agent per slide instance, running them in parallel
  3. Assembles: Merges all agent outputs into a single JSON file ready to apply to your template

This approach scales efficiently: generating 30 slides takes roughly the same time as generating 3 (since agents run in parallel, with a batch limit of 5).

The Agentic Pipeline

Phase 1: Context & Orchestration

  • Context Reading: SOLON reads files from your AI Context/ folder. Supports Excel (.xlsx, .xls), CSV, plain text, Markdown, PDF, and Word (.docx).
  • Row Filtering: If you've set filters (e.g., "PI = 28"), only matching rows are used.
  • Orchestration Call: If your flow uses repeatable slides with AI-driven grouping, an orchestrator AI call reads the context schema and identifies how many instances to create (e.g., "Create one slide per PI").
  • Slicing: For each identified instance, a slicer AI call extracts the relevant data from the context according to your slice output template.

Phase 2: Parallel Agent Generation

  • Agent Spawning: One AI agent is created per slide instance (e.g., 6 agents for 6 initiative groups).
  • Agent Context: Each agent receives the full zone schema, template, slice output rules, and its instance-specific data.
  • Parallel Execution: Agents run in batches (up to 5 concurrent). Each agent generates JSON for all zones on its slide.
  • JSON Assembly: All agent outputs are merged into a single JSON object matching your zone structure.
  • Completion Pass: If any fields are missing, targeted AI calls fill them in automatically.

Model Selection

By default, SOLON uses Claude Haiku (fast, cost-effective) for most generation tasks. If you're generating many slides, Haiku's speed is important — generating 30 slides with Sonnet might time out, while Haiku completes reliably.

For complex reasoning (e.g., if you only have 3–5 slides and need very high quality), you can edit server/config.js to switch to Sonnet, but note the trade-off: slower, more expensive, higher timeout risk.

Debugging & Logs

During generation, you can see real-time logs in the browser. After generation completes, SOLON saves debug files to flows/<flowId>/debug/:

  • ai-orchestrator-prompt.txt — Prompt sent to the orchestrator AI
  • ai-slice-instance-<N>-<slug>.txt — Data slice for each instance
  • ai-agent-prompts.txt — Prompts sent to each agent

These help you understand what the AI saw and refine your context or prompts for next time.

Error Handling & Retries

If an agent fails (e.g., due to network timeout), you can retry it individually using the UI. SOLON re-runs that agent and merges its output into the existing JSON. You don't need to re-generate everything.

Context Preparation: Best Practices

The quality of generated content depends heavily on the quality and clarity of your AI context. Here are essential tips from real-world use:

1. Pre-Calculate All Numbers

AI is notoriously bad at arithmetic. Don't ask it to sum or calculate — provide pre-calculated values explicitly in your context.

Bad: "Here are all features; count them and provide total estimate."

Good: Add a row in your spreadsheet: "Total Features: 24, Total Estimate: 450 days"

SOLON's feature catalog example uses a separate "Request A Stats" sheet with pre-calculated totals per PI and requester.

2. Use Spreadsheets (Excel/CSV) for Structured Data

Spreadsheets are the best context format. SOLON converts them to pipe-delimited text with column headers and a schema summary, which AI understands well.

Layout: One row per item, columns for all attributes (ID, Name, Status, Owner, Estimate, etc.). Use consistent, short column names.

3. Filter Strategically

If your full context is large (e.g., roadmap for the entire year), use filters to reduce what's fed to each AI agent. Example: "only generate for PI 28" filters the spreadsheet before slicing.

Filtering reduces token usage and keeps context focused, improving quality and speed.

4. Design Slice Output Templates Carefully

The slice output template tells AI how to extract data. Be explicit and unambiguous:

For REQUEST A STATS: - Look up the row where PI = "PI28" AND Requester = "Engineering" - Read the Estimate column directly (don't calculate) - If no row exists, treat estimate as 0 - Provide the result as plain text, no formatting

Clear rules prevent AI hallucination.

5. Avoid Token Overload

SOLON has a hard limit of 400,000 characters per context. For large roadmaps:

  • Use filters to feed only relevant rows to each agent
  • Split large documents into smaller, focused context files
  • Use a grouping column to deterministically partition data (e.g., "PI" column splits by quarter)

The agentic approach mitigates this: if you have 30 initiatives across multiple PIs, each agent sees only its PI's data (via filtering) and processes it independently. Total tokens scale linearly, not exponentially.

6. Use Consistent Terminology

If your context refers to a concept as "Initiative," your template zones should also use "Initiative" (not "Epic" or "Feature"). This reduces confusion and improves AI accuracy.

7. Include Descriptions & Metadata

Don't just provide IDs and names. Include descriptions, acceptance criteria, benefit hypotheses, and other details. AI uses these to reason about content, producing richer slides.

8. Validate Your Context Manually

Before running large generations, test with a single instance or a small filter. Review the generated output; if it's incorrect, check:

  • Is your context file accurate?
  • Does your slice output template match the data layout?
  • Are column names spelled correctly?
  • Did you filter the right rows?

Publishing Presentations

Presentations vs. Exports

An export is a collection of generated slides from one flow. A presentation is a curated, navigable document built from one or more exports.

Presentations add:

  • Hierarchical slide organization (folders, groups, levels)
  • Navigation sidebar with outline
  • Slide-to-slide navigation buttons
  • Optional breadcrumb trail
  • Self-contained, standalone HTML (single file or linked)

Building a Presentation Structure

In the Metadata step, you'll see a tree builder interface:

  1. Browse Exports: View all exports across your project
  2. Create Structure: Click "New Structure" to start building a presentation layout
  3. Drag Slides: Drag individual slides from exports into the tree
  4. Organize Groups: Create folder nodes to group related slides (e.g., "Q1", "Q2", "Budget Overview")
  5. Set Hierarchy: Define nesting levels (SOLON supports up to 3–4 levels of hierarchy)
  6. Reorder: Drag slides up/down to adjust order
  7. Publish: Once satisfied, publish the structure as a presentation

What You Get

When you publish a presentation, SOLON generates:

  • index.html: Main viewer with sidebar navigation and iframe-based slide display
  • slides/ folder: Individual slide HTML files, linked by the viewer
  • meta.json: Presentation metadata (name, publish date, slide count)

The viewer is fully self-contained: you can download the entire presentations/<name>/ folder and open index.html in any browser. No server required.

Navigating a Presentation

Once published, the presentation UI includes:

  • Sidebar: Full outline of all slides and groups. Click any slide to jump to it.
  • Collapse/Expand: Groups can be collapsed to hide child slides
  • Main View: Central iframe displaying the current slide with full styling and interactivity preserved
  • Navigation Buttons: "Previous" / "Next" buttons to traverse slides
  • Breadcrumb (optional): Shows current path (e.g., "Q1 > Engineering > Overview")

Versioning & Updates

Presentations are versioned by timestamp. If you regenerate slides and want to update a presentation:

  1. Create a new export from your updated generation
  2. Edit the presentation structure: remove old slides, add new ones
  3. Publish as a new version (or overwrite the old one)

Previous presentations are archived; you can always revert to an earlier version by accessing the published folder directly.

Sharing via Toolkit / SharePoint

Note: This workflow is currently manual. SOLON's "Package" feature (automated bundling) is in development. For now, use the workaround below.

The Challenge

Presentations are stored as multi-file structures (index.html + slides/ folder). SharePoint expects a single file. To share a presentation on SharePoint, you need to bundle everything into one self-contained HTML file.

Current Workaround: Manual Bundling

  1. Generate Your Presentation: Complete the workflow and publish your presentation in SOLON
  2. Bundle the Files: Use an AI tool or script to combine all files in presentations/<name>/ into a single HTML file. The script should:
    • Read index.html
    • Embed all CSS and JavaScript inline
    • Convert relative slide links to data URIs or embedded HTML
    • Output a single self-contained HTML file (~5–10 MB for typical presentations)
  3. Upload to SharePoint: Create a SharePoint page with a Script Editor web part
  4. Paste HTML: Paste the bundled HTML content into the Script Editor's editor dialog
  5. Save & View: The page now displays your interactive presentation inline

Future: Automated Packaging

SOLON is planning to integrate automated bundling into the UI. Once released, you'll see a "Package for Toolkit" button that handles all of the above in one click.

Alternative: Direct Hosting

If your organization has a web server or CDN, you can host the presentation folder directly and share the URL. This avoids bundling and keeps file sizes manageable, but requires hosting infrastructure.

Version Control & Collaboration

Projects in Git

All SOLON projects live in server/projects/shared/ or server/projects/private/. Shared projects are committed to Git; private projects are Git-ignored.

This means:

  • Team members can clone the repository and access all shared projects
  • Changes to templates, context files, and generated exports are tracked
  • You can branch to experiment with new templates or regenerations
  • Full audit trail of all modifications

Shared Projects

A shared project is synced with Git. When you:

  • Create a flow: New files appear in your working directory (unstaged)
  • Generate content: Export files are created (unstaged)
  • Publish presentations: Presentation folder is created (unstaged)

Commit these changes via Git to share with team members. They'll pull the latest and see your flows, exports, and presentations.

Private Projects

Use private projects for drafts, experiments, or local work. These are excluded from Git (see .gitignore) and never uploaded to the remote.

Convert a project from private to shared (or vice versa) via the project dashboard. SOLON auto-updates .gitignore.

Collaboration Workflow

  1. Clone: Team member clones the shared repository
  2. Install & Run: npm install && npm run dev
  3. Access Shared Projects: All shared projects appear in the landing page
  4. Open a Flow: Team member can open an existing flow and regenerate, modify, or create new exports
  5. Commit & Push: Changes are pushed to the remote; other team members pull to stay in sync

Template & Context Management

Templates and context files are committed to Git, so template improvements and updated context data propagate to all team members automatically. For example:

  • Designer updates the template HTML → commits
  • Other team members pull → next generation uses the improved template
  • Data analyst updates the roadmap spreadsheet → commits
  • Other team members pull → next generation uses fresh data

Technical Architecture

System Overview

  • Frontend: React + Vite (port 5173)
  • Backend: Node.js + Express (port 3001)
  • Data Store: Filesystem (projects folder) + Git
  • AI Provider: Cortex API (OpenAI-compatible)
  • Key Libraries: archiver (ZIP), xlsx (Excel), mammoth (DOCX), pdf-parse (PDF), node-html-parser

Frontend Architecture

The frontend is a step-based wizard (React + Hooks). All state lives in a central App component and flows down as props:

  • ProjectLandingStep: List/create projects
  • ProjectDashboardStep: View flows, create new flow
  • HtmlUploadStep: Upload template, define zones
  • HtmlRecipeStep: Generate with AI, paste JSON response
  • HtmlPreviewStep: Preview & inline-edit generated slides
  • HtmlMetadataStep: Create exports, build presentation structures, publish

Navigation is guarded: you can only advance after completing each step.

Backend Architecture

Express routes delegate to business logic modules:

  • routes/projects.js: Project & flow CRUD
  • routes/html-flow.js: Template parsing, zone selection, export management
  • routes/ai-proxy.js: Simple AI calls (recipe generation)
  • routes/opencode-agentic.js: Multi-agent orchestration (SSE streams)
  • routes/presentations.js: Presentation publishing
  • lib/: Business logic (project manager, export manager, AI client, HTML parser, etc.)

Environment Variables

PORT=3001 AI_BASE_URL=https://cortex-endpoint AI_API_KEY=your-key AI_MODEL=claude-3-5-haiku-vertex

These must be set in server/.env at runtime. Without them, generation will fail.

Key Endpoints Summary

  • POST /api/html-flow/upload-template: Parse HTML, create pending session
  • POST /api/opencode/agentic/plan: SSE stream; orchestrate and slice
  • POST /api/opencode/agentic/run: SSE stream; spawn agents, assemble JSON
  • POST /api/html-flow/apply-content: Apply JSON to template, generate output
  • POST /api/projects/:p/flows/:f/exports: Create export from generation
  • POST /api/projects/:p/presentations: Publish presentation from structure

See the Architecture Reference (in the earlier task output) for complete endpoint list.

Database & Persistence

SOLON has no database. All state is stored as JSON files on the filesystem:

  • flow.json: Complete flow state, zone selections, generations, exports
  • project.json: Project metadata
  • export.json, project.json: Export metadata and slide index
  • presentation-structures.json: Saved presentation structures

This makes state human-readable and Git-friendly. You can inspect or even edit JSON files directly.

Error Handling & Debugging

  • Browser Console: Frontend logs (React errors, API calls)
  • Terminal: Backend logs (Express logs, AI calls, file operations)
  • Debug Files: flows/<flowId>/debug/ contains AI prompts, slices, and agent logs
  • Toast Notifications: In-app status messages and error alerts

Testing

  • Unit Tests: npm test (Vitest)
  • End-to-End Tests: npm run test:e2e (Playwright)
  • Linting: npm run lint (ESLint)

Deployment

SOLON is a full-stack Node.js + React app. You can deploy to:

  • VPS/Server: Install Node.js, clone repo, set .env, run npm run dev or npm run build + production server
  • Docker: Containerize both frontend and backend (Dockerfile provided in repo)
  • Cloud Platforms: Vercel (frontend), Render, Railway (backend), or combined platforms like Heroku or AWS

The main consideration: ensure AI API credentials are securely managed (use secret management services, not hardcoded).

Summary & Next Steps

SOLON Slide Studio is a powerful tool for automating data-driven presentation generation. Whether you're creating a feature catalog, roadmap overview, or recurring status report, SOLON's agentic AI and template-based approach can save hours of manual work.

The Core Workflow: Create a project → Upload a template → Define zones → Provide context data → Generate with AI → Preview & edit → Export slides → Publish a structured presentation.

Getting Help

  • Check the debug files in your flow folder if generation fails
  • Review the browser console for API errors
  • Test with a small instance first (e.g., filter to one PI) before scaling to full generation
  • Reach out to the team if you encounter unexpected behavior (SOLON is a refined POC, and feedback helps improve it)

Known Limitations & Rough Edges

  • Toolkit Integration: Bundling presentations is manual (automated packaging coming soon)
  • Presentation UI: Some browser-side bugs in navigation and sidebar collapse (minor, doesn't affect core functionality)
  • Slice Output Templates: Currently managed as files; UI for creating/editing slice templates not yet built
  • Performance: Generating 50+ slides may take 5+ minutes (due to AI latency, not app overhead)

Future Improvements

  • Automated presentation bundling for toolkit/SharePoint
  • In-app slice output template editor
  • Enhanced UI/UX for the presentation tree builder
  • Support for additional AI models (e.g., GPT-4, custom fine-tuned models)
  • Presentation versioning & rollback
  • Team collaboration features (comments, approvals)
