Issue Type: Feature | IRM: IRM-29430 | Roadmap Initiative Group: CORE REVENUE MANAGEMENT CAPABILITIES | Roadmap Initiative: PENALTY & INTEREST | Roadmap Feature: Enhance Credit Interest Calculation to include unallocated payment segments | Requester: GTM | PI: PI28 | Estimate: 2986.9 | Feature Priority: p2 | Feature Status: Waiting Analysis Approval | Features Description: As a revenue authority officer I would like Solon RM to consider in credit interest calculation, all payments received from a taxpayer, both allocated and unallocated, so that final interest calculation is accurate and can be used in relevant processes.  

*Introduction*
This feature was originally introduced for SKV implementation of Reimbursement Interest in [https://jira.intrasoft-intl.com/browse/IRM-15910] the credits from Bill segments are considered in the calculations. To complete this interest calculation, system needs to consider all payment segments as well. 

Moving forward for the implemetation of the Tax Account in Northern countries but also for other implementations, in order to calculate interest accurately all payment segments, allocated and unallocated needs to be considered.  

Update the existing CREDIT_INT PI (Penalty & Interest) rule to calculate reimbursement interest on credit unallocated payment segments for tax account scenarios, ensures taxpayers receive proper interest on all credit amounts.

Currently, the system calculates interest only on bill segment balances. However, when for example liability amendments occur (e.g., reducing a VAT liability from €100 to €80 after a €100 payment has been made), the resulting €20 credit unallocated payment segment does not accrue interest in favor of the taxpayer. This creates an unfair situation where taxpayer funds held by the tax authority do not earn the required interest. 

To enhance understanding of this feature and facilitate analysis tasks please find below a business example as well as business cases where Solon Tax results in unallocated payment segments.

*System Requirements*
 * PI Rule Enhancement: Rule Code = CREDIT_INT (Reimbursement Interest Rule) Enhancement: Update calculation logic to include credit unallocated payment segments (next to credits on bill segments that is already implemented). 

*Business Example*

> Initial State:

VAT Liability: €100 (debit)

Payment Made: €100 (credit)

Result: Liability fully paid, balance = €0

> After Amendment:

Amendment: VAT liability reduced by €20 (becomes €80)

Payment Remains: €100 (unchanged)

Result: €20 credit unallocated payment segment

_Current Behavior:_

Interest calculated only on bill segment balances, thus €20 credit unallocated payment segment earns no interest

_Required Behavior:_

Interest calculated on bill segment balances AND credit unallocated payment segments, thus €20 credit unallocated payment segment earns monthly interest in favor of taxpayer

 

*Example Business Cases resulting in Unallocated Payment Segments*
 # Payment Suspense Scenarios, where for example invalid reference number results to a payment that cannot be allocated to a specific party role.
 # Payment File Processing Errors during automated payment file processing due to for example corrupted payment data or reference number mismatch.
 # Insufficient liability balance for full allocation as per the business example above. 
 # Payment Event State Transitions, when for example re-opening completed payment events where original segments need to be cancelled or payment event re-distribution scenarios requiring segment reallocation. 

*Documentation*
Document all the above adjustments for penalty and interest rule "CREDIT_INT".

*Introduction*

This feature describes the functional business requirements for a new single taxpayer overview page within Solon Tax, named the "Tax Account Overview". This new feature will introduce a period-centric overview. It will aggregate and present all financial transactions (FTs) related to a selected _reconciliation period(s)_ for a given revenue type registration, providing an immediate, consolidated view of that period's financial status. 

The purpose of this page is to provide a consolidated view of all Financial Transactions (FTs) associated with a specific reconciliation period for a taxpayer's Tax Account.

Currently, while Solon Tax supports the concept of a Tax Account ({{{}Skattekonto{}}}) and links Financial Transactions to reconciliation periods, there is no dedicated user interface that presents this information in a single, easily accessible view. This feature will bridge that gap, improving visibility and efficiency for Revenue Authority users managing these accounts.

As a reference countries that "Tax Account" is most prevalent in Europe, particularly in Scandinavia and the Nordics, which are global leaders in e-government. 

Currently implementation exists for the "Single Taxpayer Overview" pages, often refered to as 360 View, where a "Taxpayer Financial Overview" exists and can be used as reference. Particularly the last accordion of this page referred as " Financial Details per Financial Transaction". This is a page where all financial transactions for all revenue types are displayed and not only Tax Account that is the scope of this feature. 

*Functional Requirements*
h4. Navigation

A new redirect link shall be introduced on the *Revenue Type Registration (RTR)* page.
 * This link shall only be visible for RTRs that are configured as a "Tax Account" (i.e., where the {{isTaxAccount}} flag on the associated Revenue Type is true).
 * The link will navigate the user to the new "Tax Account Overview" page, passing the context of the current RTR and the selected reconciliation period.
 * Assumption is that the existing link from the party role will continue to function as is with the additional view in scope of this feature for the Tax Account. 

h4. Page Content and Layout

The "Tax Account Overview" page will contain the following elements:
 * *Filters:* User should be able to filter the following information:

 * 
 ** Reconciliation Period Filter
 ** From: 01/01/2026 
 ** To: 31/01/2026 or To: 31/03/2026, meaning more than one reconciliation periods can be selected. 

 
 * *"Update P&I" Button:* User should be able to see and select the "Update P&I" button where he/she can enter the "RECALCULATEFROMDATE" to run Penalty & Interest. 

 * *Financial Transactions Table:* A table displaying all Financial Transactions linked to the selected reconciliation period for the given Tax Account. The table should be read-only and provide standard functionalities like sorting, pagination and Column Configurator. The table should start with an opening balance for the period selected and end with the closing balance. 

Opening Balance of a period should be equal to the closing balance of the previous period. Opening balance should be the running balance of all financial transactions prior to date from selected in the filters. 

Closing Balance of a period is the running balance and should be equal to the Opening of that period plus any financial transaction related to that period such as debts, payments (both allocated and unallocated), interests, write offs, refunds... 

Available Columns (data) for FTs should be:  
 * Creation Date
 * Effective Date (eg. Period Reference - Period the debt/payment refers to)
 * Accounting Date 
 * Origin 
 * FT ID
 * Revenue Type 
 * FT Type 
 * Posting Reason
 * Liability Balance 
 * Running Balance

Please refer to the attached example_updated for a comprehensive view of the Tax Account Overview page.
h3. Assumptions & Dependencies
 * The Revenue Type entity has a configurable attribute to designate it as a "Tax Account."
 * The Financial Transaction entity has an attribute ({{{}reconciliationPeriodCode{}}}) to link it to a specific Reconciliation Period.

*Key features highlighted in this view:*
 * *Opening Balance:* The view starts with the "Opening Balance" showing the tax account's debt from before the selected period.
 * *Running Balance:* A column shows the balance after every single transaction, which is crucial for analysis.
 * *Detailed Descriptions:* Transactions are clearly labeled (e.g., "Payroll Tax Assessment," "Payment Received").
 * *Period Reference:* A column links a transaction back to the original tax period it relates to (e.g., Q4-2025), which is vital for an officer's investigation.
