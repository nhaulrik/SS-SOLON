Feature | IRM-12464 | CORE SUPPORTING CAPABILITIES | CASE MANAGEMENT | Add and Maintain Case - Assign and Unassign Case Workers - Effort - User | GRC | PI28 | 42 | p3 | Waiting Analysis Approval | As a revenue official, I want to assign case workers (user) that have a required profile (e.g. skills, experience and part of a certain organizational unit) to a case while also considering the availability of the suitable candidates, so that a capable worker (user) with sufficient availability can work on and complete the case. Also, I want to see how much effort is involved for each assigned case worker and for the case in total.

*Introduction*

This feature extends IRM-680 in which requirements have been documented to assign and unassign a worker to/from a case. IRM-680 includes a requirement that the user can specify the expected allocation of a case worker for a case.

This feature is related to IRM-12459 in which requirements have been documented to assign and unassign one or more resource calendars to a worker.

*Functional Requirements*
 * Eligible Workers
 ** User can see the average allocation from the case start date up until the case end date for each case worker that meets any of the worker’s profiles for the case. Average allocation can be calculated by dividing the sum of the allocated effort of a case worker from the start date up until the end date of a case by the sum of the available man-hours in that same period based on the resource calendars that are effective for the case worker in that period
 ** User can see the availability in man-hours from the case start date up until the case end date for each case worker that meets any of the case worker’s profiles for the case. Available man-hours can be calculated by subtracting the sum of the allocated effort for a case worker from the start date up until the end date of a case from the sum of the available man-hours in that same period based on the resource calendars that are effective for the case worker in that period.
 * Worker’s Assignment
 ** User can see what effort has been calculated based on the provided expected allocation for the case worker’s assignment and the resource calendar(s) that are effective for this worker from the case start date up until the case end date. Effort can only be calculated if (1) a start date and end date have been defined for the related case, and (2) an expected allocation has been provided. Period(s) within the start and end date of the case for which no effective resource calendar has been defined for the case worker are considered non-working periods. Assumption is that the effort will be recalculated if (1) the case start or end date are updated, (2) the expected allocation for that case worker for that case is updated, and/or (3) a resource calendar of an assigned case worker that falls within the period in which the case has been scheduled, is updated.
 * User can see the total calculated effort for the case. Total effort is equal to the sum of the effort that was calculated for each case worker that was assigned to the case.

Feature | IRM-20329 | NON-FUNCTIONAL REQUIREMENT COVERAGE | SECURITY | Data Security - Restrict Access to Personal Data GDPR- Admin (based on data classifications) - Part 1/4 | Other | PI28 | 1217.3 | p2 | Waiting Analysis Approval | As an admin, I want to restrict access to sensitive personal data, so that only users that have a legitimate reason to access this data are granted access and unauthorized access to this data is prevented.

*Introduction*

This feature is related to the epic IRM-3106 in which requirements have been documented to secure the information in the revenue management system from unauthorized access.

This feature is related to IRM-30322 in which requirements have been documented to classify fields  (or characteristics or categories of attachments) in the application as fields that contain non-personal, personal or sensitive personal data GDPR. However, not each field that has been classified to contain (sensitive) personal data will always contain personal data, as these fields (e.g. identifier or address fields) may also be used to record data for parties that are not natural persons. Only if the data in these fields is related to a natural person (see IRM-20315), it can be considered personal data. Exception to this rule is data that cannot be related to or has not yet been related to a specific party, such as data in forms, uploaded payment data or messages in the message store that cannot or not yet be related to a specific party. This data should be considered personal if the data has been classified as such.
Depends on [https://jira.intrasoft-intl.com/browse/IRM-20322] 

 

*Out of scope :* 
Archive db
Define classifications for financial details and relevant datasets (see [https://jira.intrasoft-intl.com/browse/IRM-23836] )

 

+Clarifications:+
classify fields  : as discussed  we mean restrict access to records based on record type and defined classification levels 

+Expected Behaviors to be available :+ 
+Visibility:+ 
Field masking, e.g. Name = *****
Record hidden , e.g. record not returned
+Search+
Record hidden, e.g., No results returned, Result excluded completely
Field masking,  e.g., Result returned but masked
+Logs+
Logs should be immutable. Information may be masked. 
 

*Functional Requirements*

*For Party , Party Roles, Attachments:*  
 * Admin can restrict access to sensitive personal data based on classification level and group of users (e.g. using user access groups).
The classification levels are defined in  [https://jira.intrasoft-intl.com/browse/IRM-20322] 

 * Admin can restrict based on classification levels, who can :
 ** _Create_
 ** _Read ( 2 modes to be able to be configured :  masked or not listing the record)_ 
 ** _Update_  
 ** _Delete_
 ** _Search_ 
 ** The above permissions must be enforceable for operations initiated via the UI, direct service calls, or APIs.
_Suggestion:  Data Access Layer Enforcement of permissions_
 **  __ It is expected that the visibility of the classified details may be configured by configurators as : masked or non visible. (see also ILs)

 * Admin should be able to remove (masking or removing record visibility)  access of records based on classification levels, from data areas of Party,  party role, Attachments

+Examples:+ 
An Admin can successfully configure a User Access Group to have "Read" access to "Personal Data" but be denied "Read" access to "Sensitive Personal Data".
When a user from that group views a list of parties, parties with only "Personal Data" are visible, while parties with "Sensitive Personal Data" are either hidden or have their sensitive fields masked, as configured.
When the same user performs a search, the results correctly exclude or mask records containing "Sensitive Personal Data".

Feature | IRM-29430 | CORE REVENUE MANAGEMENT CAPABILITIES | PENALTY & INTEREST | Enhance Credit Interest Calculation to include unallocated payment segments | GTM | PI28 | 2986.9 | p2 | Waiting Analysis Approval | As a revenue authority officer I would like Solon RM to consider in credit interest calculation, all payments received from a taxpayer, both allocated and unallocated, so that final interest calculation is accurate and can be used in relevant processes.  

*Introduction*
This feature was originally introduced for SKV implementation of Reimbursement Interest in [https://jira.intrasoft-intl.com/browse/IRM-15910] the credits from Bill segments are considered in the calculations. To complete this interest calculation, system needs to consider all payment segments as well. 

Moving forward for the implemetation of the Tax Account in Northern countries but also for other implementations, in order to calculate interest accurately all payment segments, allocated and unallocated needs to be considered.  

Update the existing CREDIT_INT PI (Penalty & Interest) rule to calculate reimbursement interest on credit unallocated payment segments for tax account scenarios, ensures taxpayers receive proper interest on all credit amounts.

Currently, the system calculates interest only on bill segment balances. However, when for example liability amendments occur (e.g., reducing a VAT liability from €100 to €80 after a €100 payment has been made), the resulting €20 credit unallocated payment segment does not accrue interest in favor of the taxpayer. This creates an unfair situation where taxpayer funds held by the tax authority do not earn the required interest. 

To enhance understanding of this feature and facilitate analysis tasks please find below a business example as well as business cases where Solon Tax results in unallocated payment segments.

*System Requirements*
 * PI Rule Enhancement: Rule Code = CREDIT_INT (Reimbursement Interest Rule) Enhancement: Update calculation logic to include credit unallocated payment segments (next to credits on bill segments that is already implemented). 

*Business Example*

> Initial State:

VAT Liability: €100 (debit)

Payment Made: €100 (credit)

Result: Liability fully paid, balance = €0

> After Amendment:

Amendment: VAT liability reduced by €20 (becomes €80)

Payment Remains: €100 (unchanged)

Result: €20 credit unallocated payment segment

_Current Behavior:_

Interest calculated only on bill segment balances, thus €20 credit unallocated payment segment earns no interest

_Required Behavior:_

Interest calculated on bill segment balances AND credit unallocated payment segments, thus €20 credit unallocated payment segment earns monthly interest in favor of taxpayer

 

*Example Business Cases resulting in Unallocated Payment Segments*
 # Payment Suspense Scenarios, where for example invalid reference number results to a payment that cannot be allocated to a specific party role.
 # Payment File Processing Errors during automated payment file processing due to for example corrupted payment data or reference number mismatch.
 # Insufficient liability balance for full allocation as per the business example above. 
 # Payment Event State Transitions, when for example re-opening completed payment events where original segments need to be cancelled or payment event re-distribution scenarios requiring segment reallocation. 

*Documentation*
Document all the above adjustments for penalty and interest rule "CREDIT_INT".

Performance , Security, Traceability, Accessibility, Usability  DODs should be fulfilled.

Feature | IRM-29513 | REFERENCE IMPLEMENTATIONS | APPLICATION PROCESSES | Support four-eye principle in Solon Tax - Approval Workflow | GRC | PI28 | 42 | p3 | Waiting Analysis Approval | As an IRM Admin, I should be able to configure workflows, that enforce four-eyes approval within business processes, ensuring that one user initiates an action and a different user validates and approves it before completion.

*Introduction* 

This feature is a result of PSD-12702 for UBM project, however it is anticipated that all implementations may use this functionality to cover different business cases. As it is described in the related PSD: 

At present, SOLON does not support double attestation (four eye principle) out of the box in human case management, which conflicts with the following requirement specified in SKV's contract (applicable to UBM as well):

 
|​3.2.2.2.1.16|​Functional|​Case Management|​In the software, it shall be possible to configure the actions that require verification (double attestation) by two different case workers during manual handling, including at least:
 - Manual registration of debiting/crediting
 - Manual modification of a transaction on a tax account
 - Manual administration of Swedish and foreign disbursement accounts
 - Manual registration and modification of a disbursement block​|​LKVFM-2.9.306|

 

UBM need the possibility to employ double attestation on human tasks, such that a change affected by the completion of the case is verified by at least 2 distinct case workers. This is for example needed for imposing sanctions on either individual people, credit institutions, or countries, where one case worker would create the case, and another one would validate the right target is being sanctioned, approve the case, turn it back to the initiator, who could then complete the task and thereby enforce the sanction. A rejection should naturally not enforce anything, but require a change in the human task from the initiator, or closing of the case altogether. 

Dependent user stories for sanctions are attached as a PNG. They are not yet created as User Stories in toolkit. Furthermore manual registration of payment methods (e.g. for international bank accounts) require double attestation. It is expected that more functional cases that require double attestation will appear. As such, a generic approach to employing double attestation on a human task flow is needed.

A workaround is being worked on that could support double attestation, but is a process that requires a lot of manual setup (see attachment). If SOLON had the capability of staging a change for entities in a human task (without persisting it), waiting for either approval or rejection, the requirements for double attestation would be a simpler task to support.

 # *Implementing Foundational Rules:* Implement an initial set of high-value architectural rules to serve as a baseline and a template for future rules. These tests will run as part of the standard CI/CD pipeline, failing the build if a violation is detected.

*Notes:* 
-General Web UI Pattern
Modern list components (whether in enterprise UIs like Microsoft Fluent, Salesforce-style grids, or custom apps) use techniques like:
Virtualized rendering (only render what’s visible in the viewport, not all records). This helps handle very large lists without freezing the UI.
Indexed filters and query limits to constrain what gets pulled from the server.

-Progressive Load with Early Results
Perform a lightweight server query that fetches the top N records (e.g., 2,000).
Display those instantly as the “preview” of the full list.
Meanwhile, the system in the background can continue gathering metadata about total matches.
This gives users a quick answer — they feel in control and the app feels fast.

-Use Virtualization
Only render rows that are visible on screen (plus a buffer).
As the user scrolls, dynamically load/render more items.
This improves performance for very large lists

Feature | IRM-38250 | CORE SUPPORTING CAPABILITIES | REFERENCE DATA | Enhance codelist items codes flexibility and historical track in Solon Tax | GRC | PI28 | 522.9 | p1 | In Design | As an RM Admin, I can update reference data with a codelist item having a code that is not unique, while its description is updated with different validity periods (no overlap), so as to comply with EU regulating updates.

*Introduction* 

*Tax Activity Codes in the EU:* There is no single "tax code" for all EU members. Instead, three main systems are used:
 * {*}NACE Codes{*}: The official statistical classification for economic activities. The current version is *NACE Rev. 2.1* (effective Jan 1, 2025). Attached in this feature for reference. 
 * {*}VAT Category Codes{*}: Standardized letters (e.g., *S* for Standard, *E* for Exempt) used in electronic invoicing.
 * {*}VATEX Codes{*}: Specific codes used to justify *VAT exemptions* based on EU Directive 2006/112/EC.
 * *Customs Codes (TARIC/CN)* For the import/export of goods, the Combined Nomenclature (CN) is used, which is an 8-digit coding system. 

In Greece, AADE (Independent Authority for Public Revenue) has proceeded with the mapping of the old activity codes (KAD 2008) to the new KAD 2025, a process which began on March 1, 2026. There is a direct correlation between Greek KAD (Activity Code Numbers) and NACE codes. KAD constitute the national version of the European NACE classification.
 * {*}Structure{*}: The first 4 characters of an eight-digit Greek KAD are fully identical to the 4 characters of the corresponding NACE code.
 * {*}2026 Harmonization{*}: AADE recently proceeded (as of 01/03/2026) with an automatic update of KADs for 1.9 million Tax IDs (AFM), in order to fully harmonize them with the new European standard {*}NACE Rev. 2.1{*}.
 * {*}Purpose{*}: This mapping ensures that economic activities in Greece are comparable to those of other EU member states for statistical and regulatory purposes.

*When a Code is Updated*

When the EU updates the NACE structure:
 * {*}Automatic Migration{*}: Most codes are automatically updated by national tax authorities (like AADE in Greece) if there is a 1-to-1 match.
 * {*}Manual Selection{*}: If one old code splits into multiple new ones, the taxpayer must manually select the most accurate one via the tax portal.
 * {*}Description Changes{*}: A code's number *can stay the same while its description* *is updated* to reflect modern technology or clearer definitions.

Validity and History
 * {*}Historical Accuracy{*}: A code must be valid for the period it was used. If you are filing an amended return for 2024, you use the code that was active in 2024, not the new 2025/2026 version.
 * {*}EU-Wide Validity{*}: This principle of historicity applies to all EU countries. While the digital portals differ (e.g., myAADE in Greece), the transition logic to NACE Rev. 2.1 is harmonized across the Union.

For Greek Tax Authority
The code *must remain valid* for the period prior to the change, as tax and statistical representation follows the principle of historicity. In practice, this means the following: * {*}Principle of Historicity{*}: For any transaction, declaration, or audit concerning the past (e.g., fiscal year 2024), the code that was in effect *at that time* is used. New codes (e.g., KAD 2025/2026) apply only to actions from their effective date onwards.
 * {*}System Maintenance{*}: AADE and Eurostat systems maintain the "historical versions" of the codes. If you need to file an amended return for a previous year, you must use the old code that corresponded to that specific period.
 * {*}Migration (Mapping){*}: The mapping performed by AADE does not "erase" the past; it creates a new entry in your Registry history. The "Registry Certificate" will show the expiration date of the old code and the start date of the new one.
 * {*}Audits and Cross-checks{*}: In the event of a tax audit for a past fiscal year, auditors examine the activity based on the description and the code that were in effect at the time of the transaction.

The current Solon Tax platform implementation requires unique codes for codelist items within the same codelist. However, EU tax regulations, particularly for tax activity codes (NACE, VAT Category, VATEX), require the ability to maintain the same code with different descriptions across different validity periods to support historical accuracy and regulatory compliance.

 * Unique code constraint for codelist items within a codelist

 * No overlap between validity periods for the same code
 * Different descriptions for the same code across different time periods
 * Historical data preservation for reporting and backdated updates
