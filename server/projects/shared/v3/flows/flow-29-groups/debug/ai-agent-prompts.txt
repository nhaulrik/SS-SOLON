The current installation procedure for SolonTax (until v2.1.0 ) relies on the manual configuration and deployment of multiple Helm charts for core and auxiliary components. This process presents several challenges:
 * *High Operational Overhead:* System administrators must manually configure multiple {{values.yaml}} files, which is time-consuming.
 * *Prone to Errors:* There is a significant risk of misconfiguration when values are not synchronized correctly across different Helm charts (e.g. service endpoints).
 * *Complex Upgrades:* Managing the upgrade sequence and dependencies between components is a manual and fragile process.

*Proposed Solution*

To address these challenges, this feature proposes the development of a new Kubernetes Operator, the {*}SolonTax Operator{*}. The operator will act as a single point of control to automate and streamline the lifecycle management (installation, configuration, and upgrades) of a SolonTax environment.

By defining a Custom Resource (CR), administrators will be able to declare the desired state of a SolonTax installation, and the operator will work to fulfill that state, abstracting away the complexity of managing the underlying Helm charts.

*Acceptance Criteria*
 # *Helm Chart Orchestration:* The operator utilizes the existing {{Revenue Management}} and {{ISS}} Helm charts to deploy SolonTax components.
 # *Selective Component Installation:* The operator's Custom Resource (CR) allows specifying the installation of:
 ** {{Revenue Management}} components only.
 ** {{ISS}} components only.
 ** Both {{Revenue Management}} and {{ISS}} components.
 # *Configuration Cascading:* A single, top-level configuration value in the operator's CR (e.g., {{{}spec.env.jdbcUrl{}}}) is correctly propagated to all relevant underlying Helm charts and their respective ConfigMaps.
 # *Configuration Overrides:* The operator's CR provides a mechanism to override specific values in individual Helm charts (e.g., setting {{spec.components.revenueManagement.deployment.apiGateway.logging.level.root }}to {{DEBUG}} should override the default log level for that component only).
 # *Dependency Management:* The operator's reconciliation loop correctly manages dependencies. For example, it must ensure a database migration Job completes successfully before proceeding with the upgrade of dependent microservice Deployments.

*Technical Requirements & Considerations*
 * *Technology Stack:* The operator must be implemented using the *Java Operator SDK* ([https://github.com/operator-framework/java-operator-sdk]) and the *Spring Boot Starter* ([https://github.com/operator-framework/josdk-spring-boot-starter]).
 * *Helm Chart Dependency Strategy:* An architectural decision must be made on how the operator will access the Helm charts it manages. The implementation should consider and evaluate:
 ** Bundling the Helm charts within the operator's container image.
 ** Fetching the charts from a remote Helm registry at runtime.
 ** A hybrid approach that supports both.

*Out of Scope*
 * The initial version of this operator will *only* manage the core *SolonTax* components ({{{}Revenue Management{}}} and {{{}ISS{}}}).
 * The management of auxiliary components (e.g., databases, message brokers, logging stacks) is not included in the scope of this work but may be considered for future iterations.

Issue Type: Feature | IRM: IRM-30730 | Roadmap Initiative Group: NON-FUNCTIONAL REQUIREMENT COVERAGE | Roadmap Initiative: MAINTAINABILITY | Roadmap Feature: Introduce helm charts dependencies | Requester: Other | PI: PI29 | Estimate: 42 | Feature Priority: p3 | Feature Status: To Do | Features Description: *Problem Statement*

The current Helm chart architecture for SolonTax does not utilize Helm's dependency mechanism. This has led to several inefficiencies:
 * *Template Duplication:* Common templates and helper logic are copied across multiple charts, leading to significant code duplication.
 * *High Maintenance Overhead:* Enhancements or bug fixes made in one chart's templates must be manually identified and ported to all other charts, a process that is both time-consuming and error-prone.
 * *Inconsistent Configurations:* The {{revenue-management-be}} and {{revenue-management-be-custom}} charts are maintained as two completely separate entities, doubling the effort required for updates and increasing the risk of divergence.

This lack of a centralized structure makes the charts difficult to maintain, scale, and audit.

*Proposed Solution*

This feature will refactor the SolonTax Helm charts to adopt a more modular and maintainable structure by introducing a common *{{base}}* chart.
 # *Create a {{base}} Helm Chart:* A new {{base}} chart will be created to serve as a common dependency for all application charts. This chart will consolidate all shared templates, helpers, and logic (e.g., for Deployments, Services, ConfigMaps, etc.). It will be designed to be flexible enough to handle both backend and frontend components.

 # *Refactor {{revenue-management-be}} Charts:* The {{revenue-management-be-custom}} chart will be refactored to use the {{revenue-management-be}} chart as a parent dependency. The custom chart will become a lightweight extension, responsible only for overriding values (like image tags) and adding configurations specific to the customized microservices. This will result in a single, unified Helm release for all {{revenue-management-be}} components.

*Acceptance Criteria*
 # *Base Chart Creation:* A new {{base}} Helm chart is created and contains common, reusable templates for standard Kubernetes resources (e.g., Deployment, Service, ConfigMap, Ingress).
 # *Dependency Adoption:* All existing SolonTax application charts (both backend and frontend) are updated to declare the {{base}} chart as a dependency in their {{{}Chart.yaml{}}}.
 # *Duplication Removal:* Duplicated template logic is removed from the application charts, which now leverage the templates provided by the {{base}} chart.
 # *{{revenue-management-be}} Refactoring:* The {{revenue-management-be-custom}} chart is refactored to be a child chart of {{{}revenue-management-be{}}}.
 # *Unified Release:* A single {{helm install}} or {{helm upgrade}} of the refactored {{revenue-management-be-custom}} chart successfully deploys both the standard {{revenue-management-be}} components and the customized components as a single Helm release.
 # *Override Functionality:* The {{revenue-management-be-custom}} chart can successfully override default values (e.g., image tags) from the parent {{revenue-management-be}} chart and inject additional properties for its specific microservices.
 # *Flexibility for Exceptions:* Application charts retain the ability to introduce their own specialized templates if a specific requirement cannot be met by the common {{base}} chart.

 * *Intrusive Instrumentation:* Backend microservices require a built-in {{elastic-apm}} Java dependency, which tightly couples the application code to a specific monitoring tool.
 * *Incomplete Coverage:* The existing setup provides no mechanism to instrument and gather telemetry from critical auxiliary components like Kafka or Keycloak, creating visibility gaps in the system.

*Proposed Solution*

This feature will modernize the SolonTax observability stack by adopting OpenTelemetry as the new standard. This will be achieved by leveraging the OpenTelemetry Operator for Kubernetes to enable non-intrusive, comprehensive data collection.

 # *APM Dependency Deprecation:* The {{elastic-apm}} JAR dependency is flagged as deprecated .
 # *End-to-End Telemetry Flow:* Traces, metrics, and logs generated by the SolonTax services are successfully collected by the OpenTelemetry Collector and correctly forwarded to and indexed in the configured Elasticsearch backend.
 # *Architectural Adherence:* The observability topology is updated to reflect OpenTelemetry best practices, with the Collector serving as the primary entry point for all telemetry data.

At present, SOLON does not support double attestation (four eye principle) out of the box in human case management, which conflicts with the following requirement specified in SKV's contract (applicable to UBM as well):

 
|​3.2.2.2.1.16|​Functional|​Case Management|​In the software, it shall be possible to configure the actions that require verification (double attestation) by two different case workers during manual handling, including at least:

 - Manual modification of a transaction on a tax account
 - Manual administration of Swedish and foreign disbursement accounts
 - Manual registration and modification of a disbursement block​|​LKVFM-2.9.306|

 

UBM need the possibility to employ double attestation on human tasks, such that a change affected by the completion of the case is verified by at least 2 distinct case workers. This is for example needed for imposing sanctions on either individual people, credit institutions, or countries, where one case worker would create the case, and another one would validate the right target is being sanctioned, approve the case, turn it back to the initiator, who could then complete the task and thereby enforce the sanction. A rejection should naturally not enforce anything, but require a change in the human task from the initiator, or closing of the case altogether.

Furthermore manual registration of payment methods (e.g. for international bank accounts) require double attestation. It is expected that more functional cases that require double attestation will appear. As such, a generic approach to employing double attestation on a human task flow is needed.

A workaround is being worked on that could support double attestation, but is a process that requires a lot of manual setup (see attachment). If SOLON had the capability of staging a change for entities in a human task (without persisting it), waiting for either approval or rejection, the requirements for double attestation would be a simpler task to support.

Issue Type: Feature | IRM: IRM-374 | Roadmap Initiative Group: REFERENCE IMPLEMENTATIONS | Roadmap Initiative: SELF SERVICE FILING & PAYMENTS | Roadmap Feature: SS Payments - Initiate Payment from Self Service Portal - User | Requester: GRC | PI: PI29 | Estimate: 507.5 | Feature Priority: p2 | Feature Status: Under Analysis | Features Description: As a taxpayer, I want to see and pay my outstanding liabilities from the self service portal, so that I can understand and meet my payment obligations towards the revenue authority.

*Introduction*

This feature describes how the taxpayer can use the taxpayer self service portal to initiate a payment to pay outstanding liabilities.

The steps the taxpayer must follow to initiate and complete a payment may vary depending on the type of integration between ICARUS and the payment service providers (a.k.a. payment gateways or PSPs). Integration types to consider are (1) hosted - taxpayer leaves self service portal to complete payment on PSP portal, (2) direct post - PSP page that is used to complete the payment is embedded in a frame on self service portal and (3) non-hosted - self service portal provides features to capture all payment details and uses an API to submit details to the PSP. This feature describes the requirements assuming the 3rd type of integration is used.

Revenue authorities can depending on the requirements provide various options on the self service portal to pay outstanding liabilities, including for example (1) pay party role, (2) pay specific tax account, (3) pay a specific filing period, (4) pay a specific pay plan, (5) pay a specific collection notice, (6) pay a specific tax return, or (7) pay a specific single assessed liability (e.g. tax, penalty or interest). This feature describes the requirements for the 3rd and 7th option.

Revenue authorities can depending on the requirements provide various payment methods on the self service portal to pay outstanding liabilities. Examples include (1) credit card, (2) debit card, (3) PayPal and other similar services, (4) M-Pesa, Paym and other similar services, (5) direct debit, (6) PDS2 and other similar services and (7) guarantees registered at the revenue authority. This feature describes the requirements for the 1st option.

Flow :

*E-Signature Workflow Overview*
 # *User Initiates Signing* (frontend)

The user clicks the *"Sign"* button.
 # *Signing Initialization Request* (backend)

A request is sent to the {*}electronic signature service{*}, which:
 * Uploads the files to the {*}signature provider{*}.
 * Initializes the signing process with a specified returnUrl, used to redirect the user back to our system after signing.
 * Returns an {*}external signing UID{*}.

 # *Popup with Embedded Signing UI* (frontend)

A popup window is opened containing an iframe.
The iframe loads the signing page using the URL that includes the *external signing UID* as a query parameter.

*_[DASKALAKIS Ioannis] Please provide, if possible, some more details on the authentication and authorization scheme that may need to be followed to facilitate loading service’s provider URL for the given UID with  the User’s  browser session._*

*_No authentication needs to be done from the browser. Only the backend IP needs to be whitelisted on the service provider and our service accepting the callback needs to be exposed and reachable by the service provider. We have started a discussion about this on another channel, still waiting for a response._*
 # *User Signs Documents* (frontend)

The user completes the signing process within the loaded iframe page.
 # *Callback and Document Handling* (backend)

Our service receives a *callback* from the signature provider when the document status changes (e.g., signing completed).
At this point, we:
 * {*}Download{*}the signed version of the document from the provider.
 * {*}Validate{*}the signed document.
 * {*}Upload{*}it via the document-management

*_[DASKALAKIS Ioannis] This, along with #3, suggests that any document exchange between our system and the signature provider will only take place via service API calls with an adapter implemented in the scope of the project. Please confirm._*

*_Yes, that is correct._*
 # *User Redirection and UI Update* (frontend)

After signing, the user sees a *success message* on the provider's page and is *redirected to the* {*}returnUrl{*}.
If we are able to detect this redirect within the popup, we can:
 * Automatically {*}close the popup{*}.
 * {*}Load the signed document{*}and its metadata in our system for the user to review.
