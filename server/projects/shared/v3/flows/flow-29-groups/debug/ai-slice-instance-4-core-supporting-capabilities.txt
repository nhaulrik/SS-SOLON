*Introduction*

This feature extends IRM-680 in which requirements have been documented to assign and unassign a worker to/from a case. IRM-680 includes a requirement that the user can specify the expected allocation of a case worker for a case.

This feature is related to IRM-12459 in which requirements have been documented to assign and unassign one or more resource calendars to a worker.

*Functional Requirements*
 * Eligible Workers
 ** User can see the average allocation from the case start date up until the case end date for each case worker that meets any of the worker’s profiles for the case. Average allocation can be calculated by dividing the sum of the allocated effort of a case worker from the start date up until the end date of a case by the sum of the available man-hours in that same period based on the resource calendars that are effective for the case worker in that period
 ** User can see the availability in man-hours from the case start date up until the case end date for each case worker that meets any of the case worker’s profiles for the case. Available man-hours can be calculated by subtracting the sum of the allocated effort for a case worker from the start date up until the end date of a case from the sum of the available man-hours in that same period based on the resource calendars that are effective for the case worker in that period.

Period(s) within the start and end date of the case for which no effective resource calendar has been defined for the case worker are considered non-working periods. Assumption is that the effort will be recalculated if (1) the case start or end date are updated, (2) the expected allocation for that case worker for that case is updated, and/or (3) a resource calendar of an assigned case worker that falls within the period in which the case has been scheduled, is updated.
 * User can see the total calculated effort for the case. Total effort is equal to the sum of the effort that was calculated for each case worker that was assigned to the case.

Flow :

*E-Signature Workflow Overview*
 # *User Initiates Signing* (frontend)

The user clicks the *"Sign"* button.
 # *Signing Initialization Request* (backend)

A request is sent to the {*}electronic signature service{*}, which:
 * Uploads the files to the {*}signature provider{*}.
 * Initializes the signing process with a specified returnUrl, used to redirect the user back to our system after signing.
 * Returns an {*}external signing UID{*}.

 # *Popup with Embedded Signing UI* (frontend)

A popup window is opened containing an iframe.
The iframe loads the signing page using the URL that includes the *external signing UID* as a query parameter.

*_[DASKALAKIS Ioannis] Please provide, if possible, some more details on the authentication and authorization scheme that may need to be followed to facilitate loading service’s provider URL for the given UID with  the User’s  browser session._*

*_No authentication needs to be done from the browser. Only the backend IP needs to be whitelisted on the service provider and our service accepting the callback needs to be exposed and reachable by the service provider. We have started a discussion about this on another channel, still waiting for a response._*
 # *User Signs Documents* (frontend)

The user completes the signing process within the loaded iframe page.
 # *Callback and Document Handling* (backend)

Our service receives a *callback* from the signature provider when the document status changes (e.g., signing completed).
At this point, we:
 * {*}Download{*}the signed version of the document from the provider.
 * {*}Validate{*}the signed document.
 * {*}Upload{*}it via the document-management

*_[DASKALAKIS Ioannis] This, along with #3, suggests that any document exchange between our system and the signature provider will only take place via service API calls with an adapter implemented in the scope of the project. Please confirm._*

*_Yes, that is correct._*
 # *User Redirection and UI Update* (frontend)

After signing, the user sees a *success message* on the provider's page and is *redirected to the* {*}returnUrl{*}.
If we are able to detect this redirect within the popup, we can:
 * Automatically {*}close the popup{*}.
 * {*}Load the signed document{*}and its metadata in our system for the user to review.

*_[DASKALAKIS Ioannis] Is this required to be a document preview functionality, or it is a direct document download?_*

*_Preview is required according to project requirements._*
