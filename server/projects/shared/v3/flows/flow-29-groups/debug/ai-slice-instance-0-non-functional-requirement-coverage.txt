Issue Type: Feature | IRM: IRM-30679 | Roadmap Initiative Group: NON-FUNCTIONAL REQUIREMENT COVERAGE | Roadmap Initiative: MAINTAINABILITY | Roadmap Feature: Vanilla Installation automation - Solon Tax kubernetes operator | Requester: Other | PI: PI29 | Estimate: 80 | Feature Priority: p3 | Feature Status: To Do | Features Description: *Problem Statement*

The current installation procedure for SolonTax (until v2.1.0 ) relies on the manual configuration and deployment of multiple Helm charts for core and auxiliary components. This process presents several challenges:
 * *High Operational Overhead:* System administrators must manually configure multiple {{values.yaml}} files, which is time-consuming.
 * *Prone to Errors:* There is a significant risk of misconfiguration when values are not synchronized correctly across different Helm charts (e.g. service endpoints).
 * *Complex Upgrades:* Managing the upgrade sequence and dependencies between components is a manual and fragile process.

*Proposed Solution*

To address these challenges, this feature proposes the development of a new Kubernetes Operator, the {*}SolonTax Operator{*}. The operator will act as a single point of control to automate and streamline the lifecycle management (installation, configuration, and upgrades) of a SolonTax environment.

By defining a Custom Resource (CR), administrators will be able to declare the desired state of a SolonTax installation, and the operator will work to fulfill that state, abstracting away the complexity of managing the underlying Helm charts.

*Acceptance Criteria*
 # *Helm Chart Orchestration:* The operator utilizes the existing {{Revenue Management}} and {{ISS}} Helm charts to deploy SolonTax components.
 # *Selective Component Installation:* The operator's Custom Resource (CR) allows specifying the installation of:
 ** {{Revenue Management}} components only.
 ** {{ISS}} components only.
 ** Both {{Revenue Management}} and {{ISS}} components.
 # *Configuration Cascading:* A single, top-level configuration value in the operator's CR (e.g., {{{}spec.env.jdbcUrl{}}}) is correctly propagated to all relevant underlying Helm charts and their respective ConfigMaps.
 # *Configuration Overrides:* The operator's CR provides a mechanism to override specific values in individual Helm charts (e.g., setting {{spec.components.revenueManagement.deployment.apiGateway.logging.level.root }}to {{DEBUG}} should override the default log level for that component only).
 # *Dependency Management:* The operator's reconciliation loop correctly manages dependencies. For example, it must ensure a database migration Job completes successfully before proceeding with the upgrade of dependent microservice Deployments.

*Technical Requirements & Considerations*
 * *Technology Stack:* The operator must be implemented using the *Java Operator SDK* ([https://github.com/operator-framework/java-operator-sdk]) and the *Spring Boot Starter* ([https://github.com/operator-framework/josdk-spring-boot-starter]).
 * *Helm Chart Dependency Strategy:* An architectural decision must be made on how the operator will access the Helm charts it manages. The implementation should consider and evaluate:
 ** Bundling the Helm charts within the operator's container image.
 ** Fetching the charts from a remote Helm registry at runtime.
 ** A hybrid approach that supports both.

*Out of Scope*
 * The initial version of this operator will *only* manage the core *SolonTax* components ({{{}Revenue Management{}}} and {{{}ISS{}}}).
 * The management of auxiliary components (e.g., databases, message brokers, logging stacks) is not included in the scope of this work but may be considered for future iterations.

Problem Statement

The current module structure of *revenue-management-be-custom* is not aligned with hexagonal architecture principles.

Hexagonal architecture is the architectural foundation of the *SolonTax* and the same principles should consistently apply to {*}be-custom{*}.

At present, the structure does not clearly separate application, domain, and adapter layers, which makes the module harder to maintain, evolve, and use as a proper extension point.

The target modular structure should consist of:
 * microservice-adapter
 * microservice-app
 * microservice-core
 * microservice-rest-api
 * microservice-dto (optional)
 * microservice-domain (optional)
 * microservice-business-specific (optional, more than one module possible)

Issue Type: Feature | IRM: IRM-30730 | Roadmap Initiative Group: NON-FUNCTIONAL REQUIREMENT COVERAGE | Roadmap Initiative: MAINTAINABILITY | Roadmap Feature: Introduce helm charts dependencies | Requester: Other | PI: PI29 | Estimate: 42 | Feature Priority: p3 | Feature Status: To Do | Features Description: *Problem Statement*

The current Helm chart architecture for SolonTax does not utilize Helm's dependency mechanism. This has led to several inefficiencies:
 * *Template Duplication:* Common templates and helper logic are copied across multiple charts, leading to significant code duplication.
 * *High Maintenance Overhead:* Enhancements or bug fixes made in one chart's templates must be manually identified and ported to all other charts, a process that is both time-consuming and error-prone.
 * *Inconsistent Configurations:* The {{revenue-management-be}} and {{revenue-management-be-custom}} charts are maintained as two completely separate entities, doubling the effort required for updates and increasing the risk of divergence.

This lack of a centralized structure makes the charts difficult to maintain, scale, and audit.

*Proposed Solution*

This feature will refactor the SolonTax Helm charts to adopt a more modular and maintainable structure by introducing a common *{{base}}* chart.
 # *Create a {{base}} Helm Chart:* A new {{base}} chart will be created to serve as a common dependency for all application charts. This chart will consolidate all shared templates, helpers, and logic (e.g., for Deployments, Services, ConfigMaps, etc.). It will be designed to be flexible enough to handle both backend and frontend components.

 # *Refactor {{revenue-management-be}} Charts:* The {{revenue-management-be-custom}} chart will be refactored to use the {{revenue-management-be}} chart as a parent dependency. The custom chart will become a lightweight extension, responsible only for overriding values (like image tags) and adding configurations specific to the customized microservices. This will result in a single, unified Helm release for all {{revenue-management-be}} components.

*Acceptance Criteria*
 # *Base Chart Creation:* A new {{base}} Helm chart is created and contains common, reusable templates for standard Kubernetes resources (e.g., Deployment, Service, ConfigMap, Ingress).
 # *Dependency Adoption:* All existing SolonTax application charts (both backend and frontend) are updated to declare the {{base}} chart as a dependency in their {{{}Chart.yaml{}}}.
 # *Duplication Removal:* Duplicated template logic is removed from the application charts, which now leverage the templates provided by the {{base}} chart.
 # *{{revenue-management-be}} Refactoring:* The {{revenue-management-be-custom}} chart is refactored to be a child chart of {{{}revenue-management-be{}}}.
 # *Unified Release:* A single {{helm install}} or {{helm upgrade}} of the refactored {{revenue-management-be-custom}} chart successfully deploys both the standard {{revenue-management-be}} components and the customized components as a single Helm release.
 # *Override Functionality:* The {{revenue-management-be-custom}} chart can successfully override default values (e.g., image tags) from the parent {{revenue-management-be}} chart and inject additional properties for its specific microservices.
 # *Flexibility for Exceptions:* Application charts retain the ability to introduce their own specialized templates if a specific requirement cannot be met by the common {{base}} chart.
