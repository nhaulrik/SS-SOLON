*Introduction* 

*Tax Activity Codes in the EU:* There is no single "tax code" for all EU members. Instead, three main systems are used:
 * {*}NACE Codes{*}: The official statistical classification for economic activities. The current version is *NACE Rev. 2.1* (effective Jan 1, 2025). Attached in this feature for reference. 
 * {*}VAT Category Codes{*}: Standardized letters (e.g., *S* for Standard, *E* for Exempt) used in electronic invoicing.
 * {*}VATEX Codes{*}: Specific codes used to justify *VAT exemptions* based on EU Directive 2006/112/EC.
 * *Customs Codes (TARIC/CN)* For the import/export of goods, the Combined Nomenclature (CN) is used, which is an 8-digit coding system. 

In Greece, AADE (Independent Authority for Public Revenue) has proceeded with the mapping of the old activity codes (KAD 2008) to the new KAD 2025, a process which began on March 1, 2026. There is a direct correlation between Greek KAD (Activity Code Numbers) and NACE codes. KAD constitute the national version of the European NACE classification.
 * {*}Structure{*}: The first 4 characters of an eight-digit Greek KAD are fully identical to the 4 characters of the corresponding NACE code.
 * {*}2026 Harmonization{*}: AADE recently proceeded (as of 01/03/2026) with an automatic update of KADs for 1.9 million Tax IDs (AFM), in order to fully harmonize them with the new European standard {*}NACE Rev. 2.1{*}.
 * {*}Purpose{*}: This mapping ensures that economic activities in Greece are comparable to those of other EU member states for statistical and regulatory purposes.

*When a Code is Updated*

When the EU updates the NACE structure:
 * {*}Automatic Migration{*}: Most codes are automatically updated by national tax authorities (like AADE in Greece) if there is a 1-to-1 match.
 * {*}Manual Selection{*}: If one old code splits into multiple new ones, the taxpayer must manually select the most accurate one via the tax portal.
 * {*}Description Changes{*}: A code's number *can stay the same while its description* *is updated* to reflect modern technology or clearer definitions.

Validity and History
 * {*}Historical Accuracy{*}: A code must be valid for the period it was used. If you are filing an amended return for 2024, you use the code that was active in 2024, not the new 2025/2026 version.
 * {*}EU-Wide Validity{*}: This principle of historicity applies to all EU countries. While the digital portals differ (e.g., myAADE in Greece), the transition logic to NACE Rev. 2.1 is harmonized across the Union.

For Greek Tax Authority
The code *must remain valid* for the period prior to the change, as tax and statistical representation follows the principle of historicity. In practice, this means the following: * {*}Principle of Historicity{*}: For any transaction, declaration, or audit concerning the past (e.g., fiscal year 2024), the code that was in effect *at that time* is used. New codes (e.g., KAD 2025/2026) apply only to actions from their effective date onwards.
 * {*}System Maintenance{*}: AADE and Eurostat systems maintain the "historical versions" of the codes. If you need to file an amended return for a previous year, you must use the old code that corresponded to that specific period.
 * {*}Migration (Mapping){*}: The mapping performed by AADE does not "erase" the past; it creates a new entry in your Registry history. The "Registry Certificate" will show the expiration date of the old code and the start date of the new one.
 * {*}Audits and Cross-checks{*}: In the event of a tax audit for a past fiscal year, auditors examine the activity based on the description and the code that were in effect at the time of the transaction.

The current Solon Tax platform implementation requires unique codes for codelist items within the same codelist. However, EU tax regulations, particularly for tax activity codes (NACE, VAT Category, VATEX), require the ability to maintain the same code with different descriptions across different validity periods to support historical accuracy and regulatory compliance.

*Current Implementation Support*

The platform currently supports:
 * Unique code constraint for codelist items within a codelist
 * Basic validity period management with valid_from and valid_to fields
 * Codelist item labels with language support

*To Be Implementation*

Allow the same code to be added for different validity periods with:
 * No overlap between validity periods for the same code
 * Different descriptions for the same code across different time periods
 * Historical data preservation for reporting and backdated updates

*Functional Business Requirements*
 # {*}Historical Code Management{*}: Enable codelist items to have the same code with different descriptions across non-overlapping validity periods
 # {*}Temporal Data Integrity{*}: Ensure no validity period overlaps exist for items with the same code within a codelist
 # {*}Backdated Processing Support{*}: Allow selection of historical codelist items when processing backdated taxpayer updates
 # {*}Regulatory Compliance{*}: Support EU tax code evolution requirements (NACE Rev. 2.1)
 # {*}Reporting Continuity{*}: Maintain historical reference data for accurate reporting across different time periods
