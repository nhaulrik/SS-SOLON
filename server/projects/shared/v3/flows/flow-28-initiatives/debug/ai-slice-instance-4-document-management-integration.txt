Issue Type: Feature | IRM: IRM-36668 | Roadmap Initiative Group: CORE SUPPORTING CAPABILITIES | Roadmap Initiative: DOCUMENT MANAGEMENT INTEGRATION | Roadmap Feature: Digital Signatures - System digital signing of PDF documents generated by bpmn process | Requester: GRC | PI: PI28 | Estimate: 1344 | Feature Priority: p1 | Feature Status: In Design | Features Description: As a revenue authority user, when the system generates a document (e.g., operational reports, letters, certificates) and the business process instructs so, I want the document to be digitally signed by the system, with all required visual elements rendered in the PDF, so that I can securely distribute officially signed documents to taxpayers and stakeholders.

This capability must be reusable, configurable, auditable, and resilient to failures.

*Background :*
In the current (as-is) systems:
Documents are generated via operational reports 
Documents may be instructed to be generated via BPMN flows.
Documents may be sent to contacts and/or stored as attachments. 
Attachments can be linked to a Document Management System (DMS) via integration.

*Assumptions :* 
*a)* Revenue authority will supplied by an External provider the digital stamp (ψηφιακή σφραγίδα) in ΡΕΜ or PKC#7 files
Then the configurations for digital signs and certificate will be used for documents signature. 
./pdf_sign_pem_multicert <input_pdf> <output_pdf> <cert.pem> <key.pem>

for our tests we need to use one from CA authority 
*b)* Any additional elements (document IDs, DMS URLs, QR codes) must be printed on the document via project-specific implementation.
{*}c){*}Attachments are integrated with a DMS per customer configuration and responsibility.
*d)* One reference configuration for one digital signature will be only will be available for testing purposes (as demo). If projects needs several signatures, they may be able to configure them in the system.  

*Out of Scope* (for now)
End-user personal digital signatures
Non-PDF document formats
Additional IDs or other document additional elements other than digital signature. (document ID, qr code)

 

 

*Functional Requirements*

*Digital Signature service* 
 * Documents can be digitally signed when instructed by a step in a BPMN flow.
 * The system must expose an API or internal service or other method  for digital signing that can be called by bpmn processes.
 * The signing service must support the signature based on customers configurations and certificate.
 * Admin may configure several signatures.  (May be able to use more than one configurations and certificate / different signatures and bpmn process should decide on which configurations and certificate to be used, but will not be any defaults available in the product).  
 * Keep Audit log for the documents' updates.

*PDF* 
 * The digital signature must be embedded in the PDF according to industry standards (e.g., PAdES).
 * The PDF must remain verifiable by standard PDF readers.

*Storage & Metadata*
 * Signed documents must be stored as attachments  (expected as per relevant integration to be synchronized with the DMS).

 * Document generated is expected to be displayed with  the relevant signature stamp and document ID  (by a reserve id method optionally to be used  in the upload  and that may be used if  needed by configurators) 

*Visual Elements on Document*
Signed documents must display a visible digital signature stamp.

*Unhappy path :*
 * Errors must be logged and be visible.   
 ** Exceptions should be thrown if document generation or digital signature process not completed successfully. 
examples: Document generation fails, Template not found, Missing parameters, Digital signing fails, Timeouts

 

*Reference implementation:* 
 * At the end of the Busreg form, generate a document (operational report with some details filled and stored in attachments linked to the form) 
or use the contact generated at the final step, that will be digital signed.
Consideration: Create the signature as a reusable subflow

*Non functional requirements*
It is expected team to get aligned with respective teams and reassure that is not creating issues or the related NFRs will be tested :
Performance , Security, Accessibility, Usability 
DODs should be fulfilled.

+Security+
Certificates and private keys must be stored securely (e.g., HSM, secure vault, encrypted storage) or other repository. 
Access to signing functionality must be restricted and auditable.

+Auditability+
Every signing action must be logged:
Who/what triggered the signing
Timestamp
Document ID
Certificate used

+Performance+
Signing should not block critical user flows beyond acceptable SLA limits.
Support async signing for large documents or batch processing.

+Configurability+
Be able to select different configurations for digital signs  to be used.
