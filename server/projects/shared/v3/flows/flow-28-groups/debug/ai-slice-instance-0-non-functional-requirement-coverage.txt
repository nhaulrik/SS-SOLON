Problem Statement

The current module structure of *revenue-management-be-custom* is not aligned with hexagonal architecture principles.

Hexagonal architecture is the architectural foundation of the *SolonTax* and the same principles should consistently apply to {*}be-custom{*}.

At present, the structure does not clearly separate application, domain, and adapter layers, which makes the module harder to maintain, evolve, and use as a proper extension point.

Since *be-custom* is intended to serve clients as a starting point for implementing their own platform customizations, the current structure does not provide a clean and consistent reference model to follow.

Restructure *revenue-management-be-custom* to follow hexagonal architecture principles and align it with the architectural standards of the main *revenue-management-be* project.

The target modular structure should consist of:
 * microservice-adapter
 * microservice-app
 * microservice-core
 * microservice-rest-api
 * microservice-dto (optional)
 * microservice-domain (optional)
 * microservice-business-specific (optional, more than one module possible)

Issue Type: Feature | IRM: IRM-30741 | Roadmap Initiative Group: NON-FUNCTIONAL REQUIREMENT COVERAGE | Roadmap Initiative: MAINTAINABILITY | Roadmap Feature: Introduce ArchUnit | Requester: Other | PI: PI28 | Estimate: 373.1 | Feature Priority: p1 | Feature Status: In Design | Features Description: *Problem Statement*

As the SolonTax codebase grows in size and complexity, it becomes increasingly difficult to manually enforce architectural guidelines and design principles ( see D0160 Programming Guidelines) during code reviews. This leads to several risks:
 * *Architectural Drift:* Over time, the code can deviate from its intended design (e.g., layered architecture, domain model purity), leading to tight coupling and reduced maintainability.
 * *Inconsistent Code Reviews:* Architectural violations can be easily missed during manual reviews, as reviewers are often focused on business logic.
 * *Increased Technical Debt:* Unchecked violations accumulate as technical debt, making future development, refactoring, and onboarding of new team members more difficult and costly.

There is currently no automated mechanism to prevent such architectural decay, making the system's long-term health dependent on tribal knowledge and manual vigilance.

*Proposed Solution*

This enabler will introduce {*}ArchUnit{*}, a free Java library for checking the architecture of your code, into our build and test lifecycle. By writing plain Java unit tests, we can codify our architectural rules and automatically verify them.

The solution involves:
 # *Integrating ArchUnit:* Add the ArchUnit dependency to the project's build configuration (e.g., Maven or Gradle).
 # *Creating an Architecture Test Suite:* Establish a dedicated test source set or package for architectural tests.
 # *Implementing Foundational Rules:* Implement an initial set of high-value architectural rules to serve as a baseline and a template for future rules. These tests will run as part of the standard CI/CD pipeline, failing the build if a violation is detected.

This will provide immediate, automated feedback to developers and act as a powerful "guardrail" to prevent architectural erosion.

*Acceptance Criteria*
 # *Dependency Integration:* The ArchUnit library is successfully added as a test-scoped dependency.
 # *Test Suite Creation:* A dedicated maven module is created to house all ArchUnit tests.
 # *CI/CD Integration:* The architectural test suite is configured to run automatically as part of the continuous integration pipeline during unit tests phase. A build must fail if any architectural rule is violated.
 # *Baseline Pass:* All newly introduced architectural tests pass against the existing codebase on the main development branch. If any pre-existing violations are found, they must be either fixed or explicitly exempted as part of this task's scope.

*Out of Scope*
 * This enabler focuses on establishing the framework and implementing a foundational set of rules. Defining an exhaustive list of every possible architectural rule is not in scope.
 * Large-scale refactoring of the existing codebase to fix numerous pre-existing violations is out of scope, unless a small, agreed-upon number of fixes are required to establish a clean baseline.

*Notes:* 
-General Web UI Pattern
Modern list components (whether in enterprise UIs like Microsoft Fluent, Salesforce-style grids, or custom apps) use techniques like:
Virtualized rendering (only render what’s visible in the viewport, not all records). This helps handle very large lists without freezing the UI.
Indexed filters and query limits to constrain what gets pulled from the server.

-Progressive Load with Early Results
Perform a lightweight server query that fetches the top N records (e.g., 2,000).
Display those instantly as the “preview” of the full list.
Meanwhile, the system in the background can continue gathering metadata about total matches.
This gives users a quick answer — they feel in control and the app feels fast.

-Use Virtualization
Only render rows that are visible on screen (plus a buffer).
As the user scrolls, dynamically load/render more items.
This improves performance for very large lists
