*Functional Requirements*
 * Admin can use the following matching criteria when defining profiles for the most suitable case workers for a specific type of case:
 ** Average Allocation: can be calculated by dividing the sum of the allocated effort of a worker within a given period (from the start date up until the end date of a case) by the sum of the available man-hours in that same period based on the resource calendars that are effective for the case worker in that period. Admin can for example define that the average allocation should be equal to or less than 0.8.
 ** Available man-hours: can be calculated by subtracting the sum of the allocated effort of a case worker within a given period (from the start date up until the end date of a case) from the sum of the available man-hours in that same period based on the resource calendars that are effective for the case worker in that period. *Introduction*

This feature extends IRM-680 in which requirements have been documented to assign and unassign a worker to/from a case. IRM-680 includes a requirement that the user can specify the expected allocation of a case worker for a case.

This feature is related to IRM-12459 in which requirements have been documented to assign and unassign one or more resource calendars to a worker.

*Functional Requirements*
 * Eligible Workers
 ** User can see the average allocation from the case start date up until the case end date for each case worker that meets any of the worker’s profiles for the case. Average allocation can be calculated by dividing the sum of the allocated effort of a case worker from the start date up until the end date of a case by the sum of the available man-hours in that same period based on the resource calendars that are effective for the case worker in that period
 ** User can see the availability in man-hours from the case start date up until the case end date for each case worker that meets any of the case worker’s profiles for the case. Available man-hours can be calculated by subtracting the sum of the allocated effort for a case worker from the start date up until the end date of a case from the sum of the available man-hours in that same period based on the resource calendars that are effective for the case worker in that period.

 ** User can see what effort has been calculated based on the provided expected allocation for the case worker’s assignment and the resource calendar(s) that are effective for this worker from the case start date up until the case end date. Period(s) within the start and end date of the case for which no effective resource calendar has been defined for the case worker are considered non-working periods. Assumption is that the effort will be recalculated if (1) the case start or end date are updated, (2) the expected allocation for that case worker for that case is updated, and/or (3) a resource calendar of an assigned case worker that falls within the period in which the case has been scheduled, is updated.
 * User can see the total calculated effort for the case. Total effort is equal to the sum of the effort that was calculated for each case worker that was assigned to the case.

*Background :*
In the current (as-is) systems:
Documents are generated via operational reports 
Documents may be instructed to be generated via BPMN flows.
Documents may be sent to contacts and/or stored as attachments. 
Attachments can be linked to a Document Management System (DMS) via integration.

*Assumptions :* 
*a)* Revenue authority will supplied by an External provider the digital stamp (ψηφιακή σφραγίδα) in ΡΕΜ or PKC#7 files
Then the configurations for digital signs and certificate will be used for documents signature. 
./pdf_sign_pem_multicert <input_pdf> <output_pdf> <cert.pem> <key.pem>

for our tests we need to use one from CA authority 
*b)* Any additional elements (document IDs, DMS URLs, QR codes) must be printed on the document via project-specific implementation.
{*}c){*}Attachments are integrated with a DMS per customer configuration and responsibility.
*d)* One reference configuration for one digital signature will be only will be available for testing purposes (as demo). If projects needs several signatures, they may be able to configure them in the system.  

*Out of Scope* (for now)
End-user personal digital signatures
Non-PDF document formats
Additional IDs or other document additional elements other than digital signature. (document ID, qr code)

 

 

*Functional Requirements*

*Digital Signature service* 
 * Documents can be digitally signed when instructed by a step in a BPMN flow.
 * The system must expose an API or internal service or other method  for digital signing that can be called by bpmn processes.
 * The signing service must support the signature based on customers configurations and certificate.
 * Admin may configure several signatures.  (May be able to use more than one configurations and certificate / different signatures and bpmn process should decide on which configurations and certificate to be used, but will not be any defaults available in the product).  
 * Keep Audit log for the documents' updates.

*PDF* 
 * The digital signature must be embedded in the PDF according to industry standards (e.g., PAdES).
 * The PDF must remain verifiable by standard PDF readers.

*Storage & Metadata*
 * Signed documents must be stored as attachments  (expected as per relevant integration to be synchronized with the DMS).

 * Document generated is expected to be displayed with  the relevant signature stamp and document ID  (by a reserve id method optionally to be used  in the upload  and that may be used if  needed by configurators) 

*Visual Elements on Document*
Signed documents must display a visible digital signature stamp.

*Unhappy path :*
 * Errors must be logged and be visible.   
 ** Exceptions should be thrown if document generation or digital signature process not completed successfully. 
examples: Document generation fails, Template not found, Missing parameters, Digital signing fails, Timeouts

Issue Type: Feature | IRM: IRM-38250 | Roadmap Initiative Group: CORE SUPPORTING CAPABILITIES | Roadmap Initiative: REFERENCE DATA | Roadmap Feature: Enhance codelist items codes flexibility and historical track in Solon Tax | Requester: GRC | PI: PI28 | Estimate: 522.9 | Feature Priority: p1 | Feature Status: In Design | Features Description: As an RM Admin, I can update reference data with a codelist item having a code that is not unique, while its description is updated with different validity periods (no overlap), so as to comply with EU regulating updates.

*Introduction* 

*Tax Activity Codes in the EU:* There is no single "tax code" for all EU members. Instead, three main systems are used:
 * {*}NACE Codes{*}: The official statistical classification for economic activities. The current version is *NACE Rev. 2.1* (effective Jan 1, 2025). Attached in this feature for reference. 
 * {*}VAT Category Codes{*}: Standardized letters (e.g., *S* for Standard, *E* for Exempt) used in electronic invoicing.
 * {*}VATEX Codes{*}: Specific codes used to justify *VAT exemptions* based on EU Directive 2006/112/EC.
 * *Customs Codes (TARIC/CN)* For the import/export of goods, the Combined Nomenclature (CN) is used, which is an 8-digit coding system. 

In Greece, AADE (Independent Authority for Public Revenue) has proceeded with the mapping of the old activity codes (KAD 2008) to the new KAD 2025, a process which began on March 1, 2026. There is a direct correlation between Greek KAD (Activity Code Numbers) and NACE codes. KAD constitute the national version of the European NACE classification.
 * {*}Structure{*}: The first 4 characters of an eight-digit Greek KAD are fully identical to the 4 characters of the corresponding NACE code.
 * {*}2026 Harmonization{*}: AADE recently proceeded (as of 01/03/2026) with an automatic update of KADs for 1.9 million Tax IDs (AFM), in order to fully harmonize them with the new European standard {*}NACE Rev. 2.1{*}.
 * {*}Purpose{*}: This mapping ensures that economic activities in Greece are comparable to those of other EU member states for statistical and regulatory purposes.

*When a Code is Updated*

When the EU updates the NACE structure:
 * {*}Automatic Migration{*}: Most codes are automatically updated by national tax authorities (like AADE in Greece) if there is a 1-to-1 match.
 * {*}Manual Selection{*}: If one old code splits into multiple new ones, the taxpayer must manually select the most accurate one via the tax portal.
 * {*}Description Changes{*}: A code's number *can stay the same while its description* *is updated* to reflect modern technology or clearer definitions.

Validity and History
 * {*}Historical Accuracy{*}: A code must be valid for the period it was used. If you are filing an amended return for 2024, you use the code that was active in 2024, not the new 2025/2026 version.
 * {*}EU-Wide Validity{*}: This principle of historicity applies to all EU countries. While the digital portals differ (e.g., myAADE in Greece), the transition logic to NACE Rev. 2.1 is harmonized across the Union.

For Greek Tax Authority
The code *must remain valid* for the period prior to the change, as tax and statistical representation follows the principle of historicity. In practice, this means the following: * {*}Principle of Historicity{*}: For any transaction, declaration, or audit concerning the past (e.g., fiscal year 2024), the code that was in effect *at that time* is used. New codes (e.g., KAD 2025/2026) apply only to actions from their effective date onwards.
 * {*}System Maintenance{*}: AADE and Eurostat systems maintain the "historical versions" of the codes. If you need to file an amended return for a previous year, you must use the old code that corresponded to that specific period.
 * {*}Migration (Mapping){*}: The mapping performed by AADE does not "erase" the past; it creates a new entry in your Registry history. The "Registry Certificate" will show the expiration date of the old code and the start date of the new one.
 * {*}Audits and Cross-checks{*}: In the event of a tax audit for a past fiscal year, auditors examine the activity based on the description and the code that were in effect at the time of the transaction.

The current Solon Tax platform implementation requires unique codes for codelist items within the same codelist. However, EU tax regulations, particularly for tax activity codes (NACE, VAT Category, VATEX), require the ability to maintain the same code with different descriptions across different validity periods to support historical accuracy and regulatory compliance.

*Current Implementation Support*

The platform currently supports:
 * Unique code constraint for codelist items within a codelist
 * Basic validity period management with valid_from and valid_to fields
 * Codelist item labels with language support

*To Be Implementation*

Allow the same code to be added for different validity periods with:
 * No overlap between validity periods for the same code
 * Different descriptions for the same code across different time periods
 * Historical data preservation for reporting and backdated updates

*Functional Business Requirements*
 # {*}Historical Code Management{*}: Enable codelist items to have the same code with different descriptions across non-overlapping validity periods
 # {*}Temporal Data Integrity{*}: Ensure no validity period overlaps exist for items with the same code within a codelist
 # {*}Backdated Processing Support{*}: Allow selection of historical codelist items when processing backdated taxpayer updates
 # {*}Regulatory Compliance{*}: Support EU tax code evolution requirements (NACE Rev. 2.1)
 # {*}Reporting Continuity{*}: Maintain historical reference data for accurate reporting across different time periods
