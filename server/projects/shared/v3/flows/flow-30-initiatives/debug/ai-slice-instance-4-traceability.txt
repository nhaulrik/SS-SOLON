Issue Type: Feature | IRM: IRM-30734 | Roadmap Initiative Group: NON-FUNCTIONAL REQUIREMENT COVERAGE | Roadmap Initiative: TRACEABILITY | Roadmap Feature: [Vanilla Installation] - Introduce OpenTelemetry observability support | Requester: Other | PI: PI30 | Estimate: 392 | Feature Priority: p3 | Feature Status: To Do | Features Description: *Problem Statement*

The current observability solution for SolonTax has several significant limitations:
 * *Intrusive Instrumentation:* Backend microservices require a built-in {{elastic-apm}} Java dependency, which tightly couples the application code to a specific monitoring tool.
 * *Incomplete Coverage:* The existing setup provides no mechanism to instrument and gather telemetry from critical auxiliary components like Kafka or Keycloak, creating visibility gaps in the system.
 * *Inflexible Architecture:* The system is hard-wired to use Elasticsearch as the sole destination for telemetry data (logs, traces, metrics). This prevents integration with other modern observability platforms or routing data to different sinks based on need.

The current approach is not aligned with modern best practices, where OpenTelemetry has emerged as the industry standard for vendor-neutral telemetry collection.

*Proposed Solution*

This feature will modernize the SolonTax observability stack by adopting OpenTelemetry as the new standard. This will be achieved by leveraging the OpenTelemetry Operator for Kubernetes to enable non-intrusive, comprehensive data collection.

The key changes include:
 # *Introduce OpenTelemetry Components:* The OpenTelemetry Operator (providing the necessary CRDs) and the OpenTelemetry Collector will be added as standard components to a vanilla SolonTax installation.
 # *Enable Non-Intrusive Instrumentation:* The existing {{elastic-apm}} dependency usage will be deprecated. Instead, instrumentation will be applied automatically at runtime using the OpenTelemetry {{Instrumentation}} Custom Resource Definition (CRD), which injects the agent without requiring code changes.
 # *Centralize Telemetry Collection:* The OpenTelemetry Collector will be configured as the central hub for all telemetry data. It will receive logs, metrics, and traces from SolonTax services and will be architected to support receiving data from auxiliary components in the future. It will then process and forward this data to the appropriate backend, such as Elasticsearch.

This aligns with the findings of the proof-of-concept from {*}IRM-20971{*}, which validated the feasibility of this approach.

*Acceptance Criteria*
 # *OTel Operator and Collector Installation:* The OpenTelemetry Operator and a default OpenTelemetry Collector instance are successfully deployed as part of the standard SolonTax installation process.
 # *Default Auto-Instrumentation:* The SolonTax and Auxilliary components are automatically instrumented via the OpenTelemetry {{Instrumentation}} CRD by default in a new installation.
 # *APM Dependency Deprecation:* The {{elastic-apm}} JAR dependency is flagged as deprecated .
 # *End-to-End Telemetry Flow:* Traces, metrics, and logs generated by the SolonTax services are successfully collected by the OpenTelemetry Collector and correctly forwarded to and indexed in the configured Elasticsearch backend.
 # *Architectural Adherence:* The observability topology is updated to reflect OpenTelemetry best practices, with the Collector serving as the primary entry point for all telemetry data.

*Out of Scope*
 * The configuration of telemetry sinks other than Elasticsearch is not included in this scope. The primary goal is to replace the collection mechanism while maintaining the existing data backend.
