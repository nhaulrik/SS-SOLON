*Representative* to *Requester*

i.e., several levels of sub-users may be able to be defined.

{color:#505f79}3) *Accounting firm* to the user  of  *Accountant employee* to delegate/assign the task{color}
{color:#505f79}or{color}
{color:#505f79}1) A (account and user) *Natural person* to a user of his/her *Guardian* (e.g. father)

4) *Accounting firm's legal representative to the*  *Accountant employee*{color}

*Functional requirements*

*Basics:* 
 * Start and End date are captured as datetime and properly defined time zone in Agreements.
 * Start and End date are captured as datetime and properly defined time zone in Agreements services.
 * Start and End date are captured as datetime and properly defined time zone in user rights.
 * The permissions should be enforced or recalled in datetime level.  (along status define in [https://jira.intrasoft-intl.com/browse/IRM-36665]) 
 * If no end date is defined the system should clearly flag in UI  "No End Date”

Note: Already available fields in Party role relationship.

*Timezone* 
 * Display datetime in the user’s configured timezone.
 * Validity must be calculated based on stored system time (not client browser time).
 * Daylight Saving Time (DST) transitions must not cause early expiration or extended validity.

*Versioning*
 * Changes should be logged in Audit log.

*Validations* 
 * End Date cannot be earlier than Start Date.
 * Start Date is mandatory.
