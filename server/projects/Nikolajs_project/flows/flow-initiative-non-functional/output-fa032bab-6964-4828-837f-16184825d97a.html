Key Epics & Features
Record Locking - Apply logic to Party record
Manages simultaneous record modifications through version control mechanism. Each Party record includes a version attribute incremented upon successful updates. System verifies submitted version matches current database version before commit, rejecting transactions on mismatch with user-friendly conflict messages.
Data Integrity improvements - Distributed Transactions
Implements error handling for synchronous operations involving distributed transactions across microservices. Covers HTTP API invocations and Kafka message consumption with appropriate retry policies, exception logging, and version reload mechanisms for Taxpayer and Registration flows.
Optimistic Locking Strategy
No long-lived database locks. Conflicts detected at commit time. UI provides reload and retry options. Batch and BPMN processes implement configurable retry policies with exception logging and version management.
Conflict Resolution Workflows
UI users receive message: "This record was modified by another user. Please reload and review the latest data." with options to reload and reapply changes. Batch/BPMN processes execute retry logic with configurable attempt counts and version reloading.
Session and Lock Management
Handles version lock release on user commit, cancellation, session timeout, and configurable timeout intervals. Addresses browser crash scenarios and multi-form concurrent updates on Party records.
NFR Validation & Documentation
Ensures Performance, Security, Accessibility, and Usability NFRs are tested and not negatively impacted. Documents all adjustments and verifies Definition of Done fulfillment across all reliability enhancements.

Admin can remove or mask access to sensitive personal data types in logs (Security, Audit, Access logs, Exceptions) based on classification levels. Audit logs generated by the system correctly mask any sensitive data fields to prevent exposure during exception handling or auditing.
